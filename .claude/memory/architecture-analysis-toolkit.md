# Analyzing this repo with LocalLLM_Code_Analysis

Verified: 2026-07-10 @ toolkit main (profile-based multi-language support; koreader preset, lua profile)

The LocalLLM_Code_Analysis toolkit (github.com/rivaborn/LocalLLM_Code_Analysis,
locally `C:\Coding\rivaborn\LocalLLM_Code_Analysis`) generates LSP-extracted
context + LLM architecture docs for this repo's Lua frontend. Extraction and a
bounded doc run (13 container widgets) were validated end-to-end on ROGSAG on
2026-07-09/10; the **full frontend doc run (~350 files, est. 15–19 h on the
homelab 3090) has NOT been run yet** — runbook below.

## One-time setup on a new machine (Windows / PowerShell 5.1)

1. Clone both repos side by side; toolkit scripts are run FROM this repo's root.
2. Python 3.10+ on PATH.
3. lua-language-server: download the `win32-x64` zip from
   https://github.com/LuaLS/lua-language-server/releases into the toolkit's
   gitignored `.tools\` (or anywhere; pass `-ServerPath <...>\bin\lua-language-server.exe`).
4. Network path to the homelab LLM gateway `192.168.1.40` (LAN or tailnet).
5. Create `.env` at THIS repo's root (gitignored, so copy this template):

   ```
   PRESET=koreader
   PROFILE=lua
   CODEBASE_DESC="KOReader e-reader application: Lua frontend on a LuaJIT/SDL runtime. Event-driven UI (UIManager window stack, string-composed event dispatch via EventListener), widget toolkit (frontend/ui/widget), document engines bridged over FFI (crengine/mupdf in koreader-base), reader feature modules (frontend/apps/reader/modules), plugin system (*.koplugin)."
   LLM_BACKEND=ollama
   LLM_HOST=192.168.1.40
   LLM_PORT=11434
   LLM_DEFAULT_MODEL=qwen3.6:27b-64k
   LLM_NUM_CTX=65536
   LLM_THINK=true
   LLM_TIMEOUT=900
   # Optional: ntfy topic URL for phone pushes + 15-min progress heartbeats
   #NOTIFY_URL=
   ```

## Step 1 — LSP extraction (free, ~10–20 min)

```powershell
cd <this repo>
powershell -File <toolkit>\llm_scripts\serena_extract.ps1 -Preset koreader -Workers 2 `
    -ServerPath <toolkit>\.tools\lua-language-server-<ver>\bin\lua-language-server.exe
```

- Output: `architecture\.serena_context\` (gitignored). Fully resumable via
  `.state\hashes.tsv` — rerun the same command after any interruption.
- `-Workers 2` needs ~4 GB free RAM (each server ~1–2.3 GB); use `-Workers 1`
  when tight. **Killed runs leak their servers** — if RAM vanishes, check
  `Get-Process lua-language-server` and `Stop-Process` the orphans.
- Expect ~50–80 "empty" files: `_meta.lua` stubs and pure-data tables have no
  symbols; that is correct. `zh_stroke_data.lua` (~1 MB data table) can hang a
  fresh server — it is legitimately skippable.
- Incoming-references coverage grows with server lifetime (lua-language-server
  resolves references best for files whose referrers it already analyzed), so
  one long uninterrupted run beats many short resumed ones. The profile
  restarts the server every 150 files because it degrades (returns empty
  symbol lists) on very long sessions.
- ROGSAG already has a full extraction (403 contexts incl. cross-file refs on
  ~100 files). To preserve it instead of re-extracting, copy the whole
  `architecture\` dir to the new machine — otherwise just re-extract.

## Step 2 — full pipeline over the frontend (LLM; est. 15–19 h)

```powershell
cd <this repo>
powershell -File <toolkit>\llm_scripts\run_pipeline.ps1 -TargetDir frontend `
    -Preset koreader -SkipSerena -Jobs 1 -Top 50
```

- `-SkipSerena` because step 1 already ran; `-Jobs 1` is forced on the ollama
  backend anyway; `-Top 50` bounds Pass 2 to the 50 highest-scoring files
  (drop the flag only if you want Pass 2 over everything — roughly doubles
  the runtime).
- The orchestrator pre-loads `qwen3.6:27b-64k` onto the 3090 via the gateway,
  runs archgen → xref → graph → overview → pass2, and writes
  `architecture\Run Report.md`. It completes a stage, lists any failed files,
  and stops after the first stage WITH failures — rerun the same command to
  resume (archgen/pass2 are hash-incremental; done files skip).
- ~2.5–3 min/file was the observed rate (thinking model). 13 container-widget
  docs already exist on ROGSAG (copied with `architecture\` if transferred;
  regenerated otherwise).
- Set `NOTIFY_URL` in `.env` for start/per-stage/complete pushes + 15-min
  progress heartbeats — recommended for an overnight run.

## Relationship to the committed event xref

`doc/events_xref.md` (regenerated by `tools/event_xref.py`) is separate and
complementary: LSP context gives per-file symbols/requires/references; the
xref maps the string-composed event dispatch (`Event:new` → `on*` handlers)
that no LSP can see.
